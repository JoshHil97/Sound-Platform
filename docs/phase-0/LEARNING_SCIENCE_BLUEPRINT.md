# Sound Academy Learning Science Blueprint

## Core Rule

Train for Sunday service performance, not lesson completion.

Every feature should answer:

- Can the learner remember what to check without being told?
- Can they hear the fault?
- Can they find the problem on the X32, Dante, Logic, Waves, P16, wireless or stream path?
- Can they fix it safely under service pressure?
- Can a mentor verify the skill in a real church context?

## Learning Loop

Every pathway should repeat this loop:

1. Learn: short concept with church context.
2. Recall: learner answers from memory before seeing help.
3. Hear: learner compares good and bad examples.
4. Do: learner performs a board, Logic, Dante or service task.
5. Diagnose: learner handles a realistic fault scenario.
6. Serve: learner demonstrates during rehearsal or service.
7. Reflect: learner records what happened and what changed.
8. Review: spaced prompts bring the skill back later.

## Method to Platform Feature Map

| Learning Method | Platform Feature | Example |
|---|---|---|
| Active recall | Cold-start questions before hints | “A mic is distorted in PFL. What do you touch first?” |
| Spaced repetition | Due review queue | Gain staging returns after 1 day, 3 days, 7 days and 30 days |
| Deliberate practice | Repeatable drills with pass criteria | Set gain on speech mic, sung vocal and choir mic |
| Scenario learning | Troubleshooting trees and Sound Lab faults | “Room has sound, stream has none. What is your first safe check?” |
| Feedback loops | Mentor sign-off and evidence cards | Mentor watches learner set gain during rehearsal |
| Service observation | Certification service log | Shadow 2 services, assist 2 services, operate 2 supervised services |
| Anti-overload | One-skill lessons and progressive disclosure | Beginner view first, advanced explanation available when needed |

## Active Recall Architecture

Every lesson should include:

- Three pre-lesson recall prompts
- Three mid-lesson checks
- Three end-of-lesson recall prompts
- One “what would you check first?” prompt
- One “explain it to a new volunteer” prompt

Example for gain:

- What is the difference between gain and fader?
- If the signal is clipped before the fader, why does lowering the fader not fix it?
- Where do you look on the X32 before changing gain?

## Spaced Repetition Architecture

Review prompts are created from:

- Lessons
- Sound Lab attempts
- Quiz results
- Mentor notes
- Incident logs
- Service Mode misses

Rhythm:

- Same day: quick recall after lesson
- Next day: practical memory prompt
- Three days: scenario question
- One week: Sound Lab or board-check prompt
- One month: service-context review
- Before serving: service-mode readiness recall

Surface prompts in:

- Dashboard: Due Reviews
- Academy: module review status
- Sound Lab: recommended repeat scenarios
- Service Mode: pre-serve checks

## Deliberate Practice Architecture

Every practical skill defines:

- Task
- Setup condition
- Expected action
- Expected sound/result
- Common mistake
- Pass criteria
- Mentor evidence required

Example:

Skill: set gain on X32 vocal channel.

Pass criteria: learner selects channel, uses PFL/input meter, asks for real performance level, sets clean gain and explains why fader is not the fix for clipping.

## Sound Lab Architecture

Sound Lab sequence:

1. Hear bad example.
2. Choose diagnosis.
3. Choose first safe board/system check.
4. Compare improved version.
5. Compare over-fixed version.
6. Reveal explanation.
7. Link to lesson, SOP and troubleshooting flow.
8. Store confidence and accuracy.

Starter scenario families:

- Clean gain vs clipped gain
- Low gain plus noise
- Feedback buildup
- Harsh EQ
- Muddy EQ
- HPF before/after
- Compression vs over-compression
- Gate chopping
- De-essing
- Limiter pumping
- Stream too dry
- Stream too roomy
- Stream too quiet
- Stream clipping

Generated browser audio is acceptable for early training. Later church-owned recordings must be reviewed, approved and tagged by sensitivity.

## Mentor Feedback Loop

Mentor feedback follows:

- Observation: mentor watches learner perform task.
- Evidence: photo, note, Sound Lab score, quiz result or service log.
- Feedback: one strength, one correction, one next action.
- Sign-off: mentor approves only when criteria are met.
- Review: skill can expire or require renewal.

Mentor sign-off required for:

- First X32 channel operation
- Routing to P16
- Dante patch checks
- Logic livestream startup
- Waves chain use
- Wireless preparation
- Troubleshooting during rehearsal
- Independent service operation

## Service Observation Requirements

| Stage | Requirement |
|---|---|
| Observe | Watch experienced operator for 2 services |
| Assist | Help with line check, wireless, P16 or Logic startup |
| Supervised Operate | Run a defined section while mentor watches |
| Independent Operate | Complete a service role without intervention |
| Reflect | Submit service notes and incident reflections |
| Renew | Re-check key skills every 6-12 months |

## Anti-Overload Rules

- Lessons should be 10-22 minutes.
- Each lesson teaches one operational decision.
- Advanced theory should be available, not forced.
- Use “first safe check” language everywhere.
- Do not show full troubleshooting trees at once on mobile.
- Sound Lab starts with 3-4 common faults before advanced faults.
- Service Mode shows only today’s checklist and urgent actions.

## Production-Ready Lesson Rule

If a lesson cannot be practiced, heard, diagnosed or observed, it is not production-ready.
